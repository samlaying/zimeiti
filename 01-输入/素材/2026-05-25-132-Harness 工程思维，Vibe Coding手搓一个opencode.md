## Harness 工程思维与实战：Vibe Coding 手搓 Agent 平台

### 概述：什么是 Harness 工程？
Harness 工程并非一个具体系统或规范，而是一种在 AI 时代，特别是面对长周期、复杂任务的“Vibe Coding”（依托 AI 进行开发）时，用于“驾驭”大模型这匹“千里马”的工程思想与实践方法。其核心目标是**确保 AI 在长时间任务中不偏离既定方向**，关键在于管理好 AI 的 **“上下文窗口”**（类比人类的短期记忆）和 **“注意力”**。

### 核心挑战：为什么 AI 会“跑偏”？
在长任务开发中，AI 会面临两大制约，这是所有 Harness 措施要解决的根本问题：
1.  **上下文窗口有限**：模型能同时处理的信息量有限，类似人类的工作记忆。任务推进时，早期信息容易被“遗忘”。
2.  **注意力稀释**：随着任务复杂化，AI 的注意力容易分散，无法持续聚焦于当前最关键的设计、编码或测试任务。

### Harness 工程的三大基本原则
为应对上述挑战，在实战中逐步形成了以下三个核心原则：

#### 1. 角色分离与对抗
-   **思想**：不依赖单个 AI 完成所有工作，而是引入多个具有不同职责的 Agent 角色，进行分工协作。这既能分担记忆压力，也能让每个角色保持高度专注。
-   **实战角色示例**：
    -   **规划与审计员**（如 Codex）：负责任务规划、技术方案设计、开发指导以及验收测试。不直接写生产代码。
    -   **后端开发者**（如 CloudCode A）：专注于后端代码的开发与修复。
    -   **前端开发者**（如 CloudCode B）：只负责前端文件，通过后端提供的接口文档进行开发。
    -   **代码管理员**（如 CloudCode C）：负责 Git 操作、版本发布和托管。
    -   **研究分析员**（如 CloudCode D）：负责分析特定代码库（如开源项目）并与现有工程对比，提供借鉴。
-   **对抗性**：审计员负责“找茬”，开发者负责“实现”和“修复”，形成制衡，保证质量。

#### 2. 文档作为跨 Agent 桥接
-   **思想**：Agent 之间不依赖人类作为“传声筒”，而是通过结构化的本地文档进行高效、充分的信息传递与对齐。
-   **关键文档类型**：
    -   **记忆文档**：每个 Agent 维护自己的 `cloud.md` 或 `agents.md`，用于记录当前进展、架构要点和待办事项，是其“长期记忆”的索引。文档过长时需归档快照。
    -   **项目契约文档**：定义项目的总目标、核心功能、数据结构等顶层设计，是全体成员的“宪法”。
    -   **协作交付文档**：
        -   **开发建议书**（Markdown）：由规划员输出，包含具体阶段的开发计划和技术方案，交由开发者阅读执行。
        -   **审计报告**（Markdown）：由审计员输出，记录验收测试中发现的问题，供开发者修复。
        -   **开放议题文档**：用于复杂任务的方案讨论，相关角色均可批注意见，产品经理（人类）也可直接标注。
    -   **接口文档**：后端维护独立的 Markdown 接口文档，供前端快速调用，避免反复阅读源码消耗上下文。

#### 3. 标准作业流程与颗粒度管理
-   **思想**：将项目拆分为版本，将版本拆分为阶段，甚至在复杂阶段内拆分子阶段。每个最小工作单元都遵循“计划 -> 开发 -> 验收 -> 修复 -> 再验收”的严格闭环流程。
-   **流程示例**：
    1.  **规划**：与审计员讨论任务，形成《开发建议书》。
    2.  **开发**：开发者阅读建议书并执行开发。
    3.  **验收**：审计员编写测试脚本，模拟真实场景进行端到端测试，输出《审计报告》。
    4.  **修复与再验收**：开发者阅读报告并修复问题，循环直至通过。
-   **颗粒度控制**：根据任务复杂度细化阶段，目的是**让 Agent 的注意力始终高度聚焦在当前最小任务单元上**，避免因任务过大导致注意力分散和产出质量下降。

### 实战项目开发历程还原
作者从一个想法开始，逐步演进工作流：
1.  **初始阶段**：与 AI 聊天形成初步规划和契约文档。
2.  **引入审计员**：为解决测试会打断开发节奏和消耗额度的问题，引入第二个 Agent（Codex）作为独立的审计与规划角色，并建立基于文档的协作流程。
3.  **分离前端开发**：后端 MVP 完成后，为保持注意力，将前端任务独立，并配备接口文档。
4.  **引入版本管理**：项目成型后，定义版本号（如 v0.1.0），并用阶段任务划分后续小版本（如 v0.2.0）。
5.  **代码管理角色化**：引入单独的 Agent 负责 Git 操作和发布。
6.  **引入研究分析角色**：为分析开源项目源码，专门启动新会话担任研究分析员，并使用开放议题文档进行集体讨论。

### 核心要点总结
Harness 工程的核心在于通过**结构化分工、文档化桥接和流程化管控**，系统地解决 AI 在长周期“Vibe Coding”中面临的上下文有限和注意力稀释问题。它要求开发者（产品经理）不能当甩手掌柜，而是要主动设计协作流程、定义角色、维护关键文档，并严格管控任务拆分与验收的颗粒度，最终像驾驭马匹一样，引导 AI 高效、稳定地抵达项目终点。作者的经历表明，这套方法并非预先设计，而是在实践中“遇山开路，遇水搭桥”逐步演化而成的，其过程本身比规则更有启发价值。
