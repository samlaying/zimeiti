# 从 Prompt 到 Skills，RAG 还行不行？

> [老纪的技术唠嗑局](https://space.bilibili.com/3546900567427713) | [BV1J1FCzrEHn](https://www.bilibili.com/video/BV1J1FCzrEHn) | 时长 50:47

## RAG 在 2026：会过时吗？
### 核心观点：RAG 不仅不会消失，反而在进化
- **Ragflow CEO (张立峰) 认为**：单纯 RAG 不足以服务 Agent，但 **R（检索）是 Agent 数据层的核心**。
    - Agent 需要的**上下文**来源于三方面：企业内部数据、工具数据、对话生成数据。
    - **Skills**（技能）比去年的设想更高一个层级，它包含了工具调用和执行计划（Plan）。
    - 未来 **“所有智能体都是 Coding Agent”**，工具调用将通过生成代码并在沙箱中执行，接口可能是文件系统。
- **OceanBase AI平台总经理 (建南) 认为**：RAG 不会消失，它和 Skills、MCP 不在一个维度。未来 Skill 和 MCP 越来越多，**仍需通过 RAG 或类似方式召回**。
    - 真正的生产应用（如生成销售材料）需要**结合应用反馈来优化 RAG**，而不仅仅是前置的切分和召回。

### 数据存储与检索策略
#### 入库（Indexing）是关键
- **建南 (OceanBase)**：**混合检索**是核心趋势。完全以非结构化切片入库，召回效率已达上限。
    - **实践建议**：在文档预处理阶段，**提取结构化数据**（如从简历、医疗文档中提取字段）。
    - **优势**：结构化数据**解决范围和准确性**，向量化检索**解决语义理解**。两者结合（混合搜索）大幅提升效率。
- **古思维 (Knowledge Labs)**：复杂RAG系统需贴近知识本质结构，并针对场景优化。
    - **通用方法**：不要将Pipeline设计得过于领域化，应按基本原则拆分，并确保数据库等基础设施能高效支持复杂结构（如Graph、Tree）的存储和检索。

#### 检索（Retrieval）层优化
- **季吴 (FastGPT)**：**标量（结构化）与向量（语义）检索结合**是当前重点。
    - **挑战**：标量字段不固定，用户可能不知道需要什么。
    - **探索方向**：**动态标量生成**，包括用户自定义扩展、模型根据场景生成标量，并存入数据库。
    - **技术借鉴**：借鉴 `Text-to-SQL` 思路，生成 SDK 代码来构建多标量关联查询的 Retrieval 流程。
- **张立峰 (Ragflow)**：技术内涵在变化。从“RAG引擎”进化到 **“上下文引擎”**。
    - **“一边推理一边搜索”** 的 Attention Engine 路线技术上吸引人，但商业上要求极低延迟（需一体机）。
    - **端到端思考**：应用是 Agent，底座是以 RAG 为基础的组件体系。数据库应作为底层支撑，**AI 中间层（如 Python）需灵活适应业务变化，并及时将成熟能力下沉至数据库**。

## Memory 与 RAG 的关系
### 核心区别：个性化与动态性
- **建南 (OceanBase)**：
    - **RAG** 处理相对**静态、客观**的知识。
    - **Memory** 为了让大模型更像人，处理**主观、个性化、动态变化**的认知和记忆（例如你记忆中随时间变化的“老王”）。
    - **不应区分该用 RAG 还是 Memory**，而应思考如何组合，共同为业务赋能。
- **古思维 (Knowledge Labs)**：
    - Memory 是**特化的 RAG**，数据源是时序消息流，对时间维度敏感（创建时间、事件时间），需考虑**遗忘机制**（时序性既是特征也是需求）。
    - Memory 可能包含不同类型（工作记忆、永久记忆），存储和检索机制比典型 RAG 更复杂。
- **季吴 (FastGPT)**：Memory 目前更多用于**C端个性化场景**，在任务流中较少被提及。可视为一种特殊 Pipeline。
- **张立峰 (Ragflow)**：
    - **技术层面**：RAG 与 Memory 在检索上无本质区别。
    - **使用层面**：RAG 入库较批量、静态；**Memory 必须实时入库**（交互生成数据需即存即搜）。
    - **现状**：Agent 平台需要 Memory，但其**区域定义和最佳实践仍在快速演变**，尚未完全收敛。OceanBase 已在其产品中集成 Memory（如 `PowerMemory`）。

## Skills 的价值与实践
### Skills 是 RAG 的增强与最佳实践载体
- **Skills 增强 RAG**：
    - **建南 提出“用 Skill 做 RAG”**：针对不同业务场景，将 RAG 的检索参数（如相似度阈值、TopK）配置为 Skill，让 Agent 根据召回结果动态调整，成为**动态最佳实践**。
- **Skills 应用场景与建议**：
    - **FastGPT (季吴)**：
        - 交付团队利用 Skills 沉淀**问题排查、系统运维、日志分析**等经验（已积累20+个）。
        - Skills 更像**自然语言定义的工作流**，对非开发人员友好。
    - **Knowledge Labs (古思维)**：维护基于 Skill 的 CLI 插件。
        - **优势**：Skill 可打包 CLI/Script，能力更强大（支持 Shell Pipeline），能在一个回合内完成复杂操作。
        - **调试经验**：用最聪明的 Agent 做端到端评估以获知改进点；关注不同模型对复杂工具类型定义的解析差异。
    - **OceanBase (建南)**：
        - Skills 分为**工具类**（提升个人/工程效率）和**业务类**（特定领域最佳实践）。
        - OceanBase 团队已内部沉淀大量 Skills，并计划未来开放更多与数据库相关的 Skills。

## 总结与展望
- **RAG 依然至关重要**：它正从单纯的检索组件，进化为支撑 Agent 的**上下文引擎**，核心是检索（Retrieval），其数据源（数据库、文件系统、Memory）和方式（向量、标量、混合、代码生成）在不断丰富。
- **数据管理成为核心**：高质量的 RAG 依赖精细的数据**预处理（提取结构化信息）** 和**混合检索策略**，以平衡准确性与语义理解。
- **Memory 是个性化扩展**：它处理动态、主观的记忆，与静态 RAG 知识库互补，共同构建更人性化的 Agent 上下文。
- **Skills 是最佳实践的载体**：它不仅能封装工具调用，更能将复杂的 RAG 检索策略、业务逻辑封装为可动态调整的“技能”，是提升 Agent 能力和效率的关键。
- **未来趋势**：端到端的优化（从应用反馈到数据层）、数据库能力的下沉（支持混合搜索、Tensor搜索等更丰富索引）、以及标准化且强大的 Skills 生态，将是 2026 年及之后的发展方向。
