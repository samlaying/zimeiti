## Agent系统质量评估方法论
在Agent系统的设计与迭代中，依赖感觉判断改进效果是不可靠的。本章旨在建立一套基于数据的质量评估方法，将“感觉好用”转变为“数据证明变好”。

### 一、Agent质量的三个核心评估维度
Agent的质量并非单一指标，至少包含三个独立维度：
1.  **行为正确性**：关注**过程**而非结果。衡量Agent在压力下是否严格遵守了既定规则和操作序列。一个Agent可能因偶然因素得到正确结果，但执行路径是错误的。
2.  **结果质量**：关注**输出**是否符合预期。此维度最直观，但易陷入主观陷阱，必须转化为可机器验证的客观条件。
3.  **可靠性**：关注**一致性**。检验同一输入在多次执行中表现是否一致，以及在边界条件（如长对话、高压力、异常输入）下是否发生退化。

### 二、各维度的评估方法与工具
#### 2.1 行为正确性评估
- **核心方法**：压力测试。不直接问Agent能否完成任务，而是问在最大压力下是否仍遵守规则。
- **量化指标**：在N次压力场景中，遵守规则的比例。
- **辅助指标（Andre Capacity提供）**：可从历史记录统计，无需专门设计测试场景。
    - `Diff`中不必要的改动行数。
    - 代码重写次数。
    - 问题是在实现之前还是之后提出。
- **示例工具**：Superpowers项目的压力测试。

#### 2.2 结果质量评估
- **核心方法**：客观化验收。将“结果好不好”转化为一系列**可机器验证**的条件。
- **设计原则**：
    - 条件必须可机器验证。
    - 覆盖正向条件（如功能正确）和保护条件（如不破坏原有功能）。
    - 包含集成验证（如代码编译通过）。
- **示例工具**：GOOS的OpenModelGym。为Agent提供具体任务（如“给UserStruct添加DisplayName方法”），并定义明确的验收条件（文件是否存在、方法名匹配正则、代码可编译、原功能保留）。

#### 2.3 可靠性评估
- **核心方法**：最差情况优先。不取平均值，而是运行多次，用**最差结果**代表可靠性。
- **具体操作**：每个测试场景默认运行三次，保留最差结果。任何一次失败即标记为失败。
- **目的**：捕获偶发性故障和边界退化，避免平均值掩盖问题。
- **示例工具**：GOOS OpenModelGym采用此策略。

### 三、将测试驱动开发（TDD）映射到Skill文档
**核心理念**：Skill文档是一种给AI执行的特殊代码，可以用TDD的思想进行质量保证。

#### 3.1 TDD与Skill开发的映射关系
| 代码TDD概念 | 对应到Skill开发 |
| :--- | :--- |
| 测试用例 | **压力场景** |
| 生产代码 | **Skill文档本身** |
| 测试失败 | **没有Skill时，Agent违规** |
| 测试通过 | **有Skill时，Agent在压力下遵守规则** |
| 重构 | **发现新违规方式后，补充Skill反驳内容** |

#### 3.2 TDD4Docs（为文档做TDD）的三个阶段
**Red阶段（基线测试）**：
- **操作步骤**：
    1.  构建三个以上的组合压力场景（如时间紧迫+沉没成本+权威施压+疲劳叠加）。
    2.  在**没有**相关Skill的情况下，让Agent处理这些场景。
    3.  **逐字记录**Agent的违规行为、具体措辞、触发违规的压力类型。
- **重要性**：此步骤绝对不能跳过。它能揭示AI在自然状态下**真正**会使用的借口和绕过方式，避免Skill文档针对的是“想象中”的问题。
- **结果**：基线测试的违规清单即为Skill的“需求文档”。

**Green阶段（编写最小Skill）**：
- **原则**：只针对基线测试清单中记录的真实问题写Skill，不多写。
- **反直觉现象**：Skill越长，AI的合规率可能反而越低。过长的Skill会导致AI只记住最突出的部分，并给模糊解读提供更多空间。
- **验证**：继续运行相同的压力场景。如果Agent仍违规，说明Skill内容不够精准，需反思是否抓到了问题的本质，而非简单增加字数。

**Refactor阶段（迭代优化）**：
- **操作**：继续运行压力场景，寻找AI新的绕过方式。
- **典型发现及处理**：
    1.  **精神vs字面逃避**：AI声称遵守“规则精神”但认为字面规则不适用。**处理**：在Skill中加入“违反规则的字面含义就是违反规则的精神”。
    2.  **模糊措辞被重新解读**：如将“尽量避免”解读为“可以在必要时违反”。**处理**：用“never”、“always”等无歧义词替换。
    3.  **Description摘要替代正文**：AI从Description获取足够信息，不再认真读正文。**修复**：Description只写触发条件，绝不写流程细节。

### 四、不同类型Skill的测试重点
| Skill类型 | 测试方法 | 关键测试点 |
| :--- | :--- | :--- |
| **纪律执行型** | 多重叠加压力场景 | 必须在多重压力（如时间+沉没成本+权威）叠加下测试，单一压力通常能通过。通过标准：在最大叠加压力下仍遵守规则。 |
| **技术型** | 新场景的应用验证 | 给出Skill未明确提到的新场景，观察AI能否正确应用。通过标准：正确识别适用场景，处理边界情况。 |
| **模式型** | **必须同时测正例和反例** | 测试AI能否识别**不该**使用该模式的场景。例如，测试“并行Agent”模式时，需包含“任务间有依赖”、“任务数过少”、“共享资源冲突”等反例。 |
| **参考型** | 检索+应用的组合场景 | 测试AI能否找到正确的API，并生成可运行的代码，而非仅仅知道API存在。 |

### 五、评估基础设施与内化流程
#### 5.1 可用的观测工具对比
- **Inno Callback**：提供节点级（On Start, On End等）的精确数据（时间、输入输出快照）。优势：精细，可按需订阅事件，开销低。适合性能基线与行为统计。
- **Hermes Trajectory**：将完整Agent执行过程保存为ShareGPT格式JSON。包含完整对话历史。优势：数据详尽，适合深度分析和微调。
- **Deerflow + LangSmith**：为LangGraph执行注入元数据，在LangSmith中提供可视化Trace。优势：支持并排对比两次执行，适合Prompt调试和个案分析。

#### 5.2 将评估内化到开发流程
评估应是持续过程，不同改动触发不同评估：
- **新写一个Skill**：运行完整的TDD（Red-Green-Refactor）流程（约30-60分钟）。
- **修改一个Skill**：运行该Skill及其语义相邻Skill的压力场景（约15-30分钟）。
- **模型版本更新**：运行**全量**压力场景库（成本最高，约2-4小时，但不可跳过）。
- **添加新工具**：运行工具调用正确性测试及全局核心场景。
- **每周例行**：随机抽样20%场景进行持续监测。

#### 5.3 解读回归测试的信号
- **意外的违规**：改了A，B的合规率下降。说明A、B内容可能冲突或共享背景信息被改变。
- **意外的遵守**：改了A，不相关的B合规率上升。可能意味着B之前在隐性依赖A的内容，这是一个架构问题。
- **模型版本更新**：是常见的行为变化来源，即使Skill不变，模型“个性”变化也可能导致行为改变。

#### 5.4 根据评估结果指导下一步行动
- 合规率下降 + 发现新借口 → **进入Refactor阶段**，在Skill中补充反驳。
- 合规率下降 + 无新借口 → 可能Skill因过长被部分内化，**考虑精简内容**。
- 结果质量下降 + 合规率正常 → 问题在任务理解层，需**改Prompt而非改Skill**。
- 可靠性下降（方差变大） → 检查长对话中的上下文退化，可能需**引入压缩策略**。

### 六、零成本起步建议
如果目前没有任何评估体系，可以从成本最低的**Andre Capacity三个指标**开始：
1.  **Diff干净度**：从Git History统计不必要改动的行数比例。
2.  **重写次数**：从Code Review记录统计因过度复杂化而重写的次数。
3.  **问题提出时机**：从对话记录统计问题是在实现前还是实现后提出。
这些指标可从现有历史记录中统计，无需额外设计测试，能立即开始量化Agent的工程成熟度。

## 总结
本章系统性地阐述了如何用数据驱动Agent系统的质量改进。核心是建立**行为正确性、结果质量、可靠性**三个维度的评估体系，并借鉴**测试驱动开发（TDD）** 的思想来规范Skill文档的编写与验证（TDD4Docs）。通过**压力测试基线**记录AI的真实违规模式，从而编写**精准、简短**的Skill进行反驳，并在**迭代中**发现和关闭新的漏洞。最后，通过将评估**流程化**、利用**观测工具**收集数据，并根据结果**诊断问题**，形成“评估-改进”的闭环，推动Agent开发从依赖感觉走向工程化成熟。
