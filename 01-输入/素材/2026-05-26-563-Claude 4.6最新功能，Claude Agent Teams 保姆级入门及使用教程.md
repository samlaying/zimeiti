## Claude Agent Teams 核心知识笔记

### 一、 Agent Teams 概念与发展历程
Agent Teams 是 Claude Code (版本4.6) 推出的一个实验性新功能，它允许用户创建一个由多个AI代理组成的团队，以协作方式完成复杂的开发任务。理解它需要将其置于与AI对话模式的发展历程中进行对比。

#### 1. 独行侠时代（普通模式）
*   **特点**：只有用户与一个“主代理”进行一对一的串行对话。
*   **优点**：简单直接。
*   **缺点**：
    *   **上下文窗口有限**：所有交互都占用主对话的上下文，容易导致信息被压缩。
    *   **串行执行**：只能按顺序完成一件事，无法并行处理。

#### 2. Sub Agents (子代理) 时代
*   **特点**：可以将特定任务（如撰写PRD、测试代码）分配给独立的“子代理”执行。
*   **优点**：
    *   **上下文隔离**：子代理在自己的独立上下文中工作，完成后只返回结果摘要，不影响主对话上下文。
    *   **并行执行**：可以同时运行多个子代理处理不同任务。
    *   **模型选择灵活**：可根据任务要求为不同子代理指定不同模型。
*   **缺点**：各个子代理之间**完全独立，无法沟通协作**。

#### 3. Agent Teams (代理团队) 时代
*   **特点**：除了具备子代理的所有优点外，关键升级在于**团队成员之间可以互相发消息沟通**，并有一个“Team Leader”（队长）负责任务分配、协调和消息转发。
*   **类比**：就像一个真实的工作团队，前端、后端、测试等角色为了同一个产品目标而协作沟通。
*   **核心优势**：实现了真正的**团队协作**，适合处理一个复杂大任务中相互关联、需要沟通的多个子任务。

### 二、 Sub Agents 与 Agent Teams 的核心区别
这是理解何时使用哪种模式的关键，务必掌握：

| 特性 | Sub Agents (子代理) | Agent Teams (代理团队) |
| :--- | :--- | :--- |
| **核心区别** | **代理之间零交流**，各自独立完成任务。 | **队员之间可以沟通**，共享上下文和信息。 |
| **协作关系** | **独立个体**，如同老板派不同员工做完全不相关的独立任务。 | **协作团队**，如同公司内为了同一产品目标而紧密合作的不同部门。 |
| **适用场景** | 任务间**无依赖、无交互**。例如：同时查资料和审代码。 | 任务间**有依赖、需协作**。例如：前端开发需与后端接口联调，测试需等待开发完成。 |
| **资源与上下文** | 各自拥有独立上下文，用完即销毁。 | **每个队员共享相同的项目上下文**（如 `CLAUDE.md`, MCP描述），因此存在上下文冗余。 |

**选择口诀**：如果子任务之间**需要沟通、是同一目标的不同阶段**，用 **Agent Teams**；如果子任务**完全独立、无需交流**，用 **Sub Agents**。

### 三、 Agent Teams 配置与使用方法

#### 1. 开启方式 (两种)
*   **临时开启（推荐初试）**：在终端中设置环境变量。
    ```bash
    export ANTHROPIC_EXPERIMENTAL_AGENT_TEAMS=1
    ```
*   **永久开启**：将配置写入 `~/.claude/settings.json` 文件。
    ```json
    {
      "experimental": {
        "agentTeams": true
      }
    }
    ```
    *注意：如果文件不存在，可新建一个。*

#### 2. 使用终端
*   可以使用系统自带终端、VS Code集成终端等。
*   **推荐终端**：使用支持分屏的 `tmux` (主要支持Mac/Linux)，可以同时查看多个队员的实时输出。
    *   在普通终端中，使用 `Shift + PageUp` 和 `Shift + PageDown` 切换查看不同队员的状态。

#### 3. 操作步骤与关键词
1.  配置好开启选项后，启动 Claude Code。
2.  在对话中，使用关键词**触发**创建团队。例如：“`创建一个Agent Teams来开发用户登录功能`”。
3.  Claude 会引导你定义**队长**（负责协调）和**队员**（如前端开发、后端开发、测试工程师）。
4.  团队会创建一个**任务列表**，并自动处理依赖关系（如测试队员会进入阻塞状态，等待开发完成）。
5.  队员之间可以**互相发送消息**进行协调（如前端向后端请求接口）。
6.  所有任务完成后，队长汇总结果，并自动进行**团队资源清理**。

### 四、 实战示例与观察
UP主以开发一个MVP网站功能为例，展示了Agent Teams的工作流。
*   **团队构成**：队长 + 前端开发工程师 + 后端开发工程师 + 测试工程师。
*   **关键流程**：
    *   并行启动多个Claude实例（每个队员一个）。
    *   前端在开发过程中，会主动向后端**发送消息**请求API接口。
    *   测试工程师在开发完成后，开始执行测试用例。
*   **模型消耗**：Agent Teams极其消耗Token。UP主尝试使用Kimi模型很快用完额度，后切换至质谱模型，并建议慎用昂贵的Opus 4.6模型。

### 五、 使用缺点与关键建议

#### 1. 主要缺点
*   **Token消耗巨大**：每个队员都是一个独立实例，消耗量倍增。
*   **上下文冗余**：相同的项目上下文（如CLAUDE.md）在每个队员的实例中加载一遍。
*   **文件冲突风险**：如果任务划分不清晰，多个队员可能同时编辑同一文件，造成冲突。
*   **任务规划难度高**：需要精心划分队员职责和任务依赖关系，否则效果可能不如串行执行。

#### 2. 最佳实践与使用建议
*   **适合的场景**：
    *   **探索与辩论**：让多个队员从不同角度分析同一问题，得出综合结论。
    *   **多维度并行分析**：队员负责不同但不冲突的维度（如UI设计、API设计、数据库设计）。
    *   **从0到1开发MVP**：任务模块划分清晰（前端/后端/测试），彼此依赖少，能充分发挥并行协作优势。
*   **不适合的场景**：
    *   **串行依赖强的任务链**：队员A必须完成，队员B才能开始。此时效率低于串行。
    *   **微小简单的任务**：不要为了“换一个灯泡”而召开“董事会”，杀鸡用牛刀，极度浪费资源。
*   **总体建议**：Agent Teams是实验性功能，适合探索和特定场景。对于常规开发，更推荐先用好功能成熟的 **Sub Agents** 来提升效率。使用Teams时，**务必提前做好清晰的模块和任务规划**，是成功的关键。

### 核心要点总结
Claude Agent Teams 是相比 Sub Agents 的一次重大升级，其核心是引入了**队员间的实时沟通与协调机制**，模拟了真实团队协作。它非常适合**需要并行处理且存在交互依赖**的复杂开发任务（如完整MVP开发），能通过分工协作提升效率。然而，其高昂的Token消耗、上下文冗余以及潜在的**文件冲突风险**，要求使用者必须具备**精细的任务规划和队员分工能力**。对于大多数独立任务，Sub Agents仍是更高效轻量的选择；只有在明确需要“团队作战”时，才应考虑启用Agent Teams，并谨记“不要为换灯泡开董事会”的原则。
