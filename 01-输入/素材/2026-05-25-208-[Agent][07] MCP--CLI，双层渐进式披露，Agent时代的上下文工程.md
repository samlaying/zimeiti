# [Agent][07] MCP-->CLI，双层渐进式披露，Agent时代的上下文工程

> [偷星九月333](https://space.bilibili.com/349950942) | [BV1HxXGBBESv](https://www.bilibili.com/video/BV1HxXGBBESv) | 时长 17:22

## 核心问题：MCP服务的冗余信息
1.  **现状描述**：一个MCP（Model Context Protocol）服务通常封装了多个工具。
2.  **当前操作**：通过大模型使用MCP服务时，需要将该服务下**所有**工具的完整描述（包括名称、参数、参数格式、用途等）一次性加载到模型的上下文中。
3.  **关键矛盾**：用户执行一个具体任务时，通常只需要调用其中**2到3个**工具。加载所有工具描述（可能达几十甚至上百个）会产生大量当前任务**完全用不到的冗余信息**。
4.  **引发的问题**：
    *   **成本增加**：冗余信息会消耗大量的Token，增加API调用成本。
    *   **模型偏移**：冗余信息可能成为“噪音”，干扰大模型对用户意图的精准理解，导致模型性能下降或产生不准确的输出。

## 解决方案：转向CLI命令行操作
1.  **CLI的优势**：命令行工具（CLI）采用**渐进式披露**的信息架构。
2.  **核心机制**：`--help` 参数是原生的渐进式披露方式。
    *   **初始状态**：模型执行命令时，初始上下文只需包含命令的**基本描述和核心参数**，信息非常精简。
    *   **按需获取**：当模型需要了解更详细的功能或特定参数的用法时，可以主动查询 `--help` 或相关子命令的帮助信息，从而获取**按需展开**的详细信息。
3.  **优势对比**：这种方式确保了模型上下文中**始终只保留当前步骤最相关的信息**，避免了MCP服务式的“信息全量加载”问题。

## 核心思路：从“全量预加载”到“渐进式披露”
1.  **信息加载策略的转变**：
    *   **MCP模式**：**预加载**所有可能的工具信息，无论当前是否需要。
    *   **CLI模式**：**按需加载**工具信息，只在需要时获取相关部分。
2.  **本质**：这是一种针对大模型上下文窗口的优化工程，通过精简初始上下文、保留按需扩展的能力，在**效率、成本和准确性**之间取得更好的平衡。

## 总结
视频的核心观点是，传统的MCP服务将全部工具描述一次性塞入上下文的做法，在针对具体任务时存在严重的冗余问题，既浪费Token成本，又可能损害模型性能。转向CLI命令行操作形式，利用其 `--help` 等机制实现信息的**渐进式披露**，是一种更高效的上下文工程策略。这确保了模型在执行任务的每一步都只处理最相关的信息，从而提升了智能体（Agent）的工作效率和可靠性。
