## 封号机制分析
Claude Code通过特定标识追踪用户，可能导致封号，但源代码中未直接收集硬件指纹如MAC地址、CPU型号等。

### 追踪标识类型
- **User ID**：永久保存，随机生成的64位16进制字符串，用于跨会话追踪，存储在特定文件中。
- **Anonymous ID**：永久存储在设备上，作为备用追踪ID。
- **Account ID**：永久存储，绑定用户账号，通过OAuth登录关联身份。
- **邮箱**：永久存储，用于身份关联。
- **仓库哈希**：按仓库进行追踪，存储项目特定信息。

### 如何查看和删除追踪标识
- **查看当前追踪ID**：使用命令 `claude --diagnostics` 或类似脚本查看存储的标识。
- **删除追踪标识**：运行脚本删除User ID、Anonymous ID等文件内容，下次启动时生成全新ID，断开历史关联。
- **清空遥测数据**：使用三条命令清除遥测内容，防止数据上传分析。
- **清除OAuth关联**：通过脚本查看并清空OAuth信息，断开账号关联。
- **完全重置**：执行一系列命令完全重置Claude Code保存的信息，等同于新用户。

### 防止封号的建议
1. **禁用非必要网络流量**：在配置文件中添加代码，禁用分析上报。
2. **使用第三方云后端**：替代默认后端，减少追踪。
3. **创建定时脚本**：定期清理追踪数据、设备ID和遥测内容。
4. **重置设备ID**：使Anthropic无法关联历史数据，避免因过往封号被识别。

## 模型偷换分析
源代码分析结论：Claude Code不存在偷换模型的行为，主对话模型完全由用户控制。

### 结论：不存在偷换模型
- 主对话模型由用户手动设置（如使用 `--model` 参数指定Opus 4.6），在对话中保持不变，源代码中无偷换功能。
- 后台使用弱模型Hiku（如Summit模型）执行辅助任务，如配合检查或摘要生成，但不替代主对话模型。
- 过载时可能从强模型切换到弱模型，但触发条件极其严格，可忽略不计。

### 后台模型使用
- Hiku模型用于后台任务，不出现在主对话中，确保主对话性能。
- 在“Effort降级”场景中，不换模型而是降低思考级别（如从max到high），当模型不支持max时发生。

### 如何验证模型
- 通过命令行查看当前模型设置，确认是否与预期一致。
- 监控对话响应，检查模型性能变化；感觉变傻时可能源于上下文压缩或其他因素，非客户端偷换。
- 使用诊断工具验证模型类型和会话参数。

## 上下文压缩算法
Claude Code采用四层递进压缩体系，比OpenAI的上下文管理更优，能保留早期对话细节。

### 四层递进压缩体系
- **第一层：微压缩**：纯规则操作，不调用大模型，清理旧工具输出结果，保留语意信息。
- **第二层：自动压缩**：基于算法自动压缩上下文，具体代码和算法在源代码中实现。
- **第三层：传统压缩**：使用传统方法进一步压缩数据。
- **第四层：session memory压缩**：压缩会话记忆，优化长期对话管理。
- **核心原则**：优先使用廉价规则操作延迟高昂的大模型调用，提高效率。

### 与OpenAI的对比
- OpenAI上下文管理较差，长对话中易遗忘早期细节。
- Claude Code通过四层压缩确保多轮对话中记忆早期内容，提升用户体验。

## 记忆系统架构
Claude Code采用三层渐进知识管理体系，支持长期记忆和团队协作。

### 三层记忆架构
1. **绘画记忆**：存储单次会话中的临时信息。
2. **持久记忆**：长期保存用户偏好和项目知识，以markdown文件格式存储。
3. **团队记忆**：支持团队协作，共享知识和进展。

### 持久记忆格式
- 文件格式为markdown，包含四种记忆类型：
  - 用户角色、目标、知识背景。
  - 用户对工作方式的指导。
  - 项目进展、目标和事件。
  - 外部系统的指针。
- 前两种类型类似OpenAI的记忆系统，但Claude Code更结构化。
