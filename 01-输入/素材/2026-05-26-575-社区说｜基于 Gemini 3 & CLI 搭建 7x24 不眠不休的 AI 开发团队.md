# 社区说｜基于 Gemini 3 & CLI 搭建 7x24 不眠不休的 AI 开发团队

> [GDG开发者社区](https://space.bilibili.com/543767670) | [BV16EFKzeEQE](https://www.bilibili.com/video/BV16EFKzeEQE) | 时长 44:11

## 一、引言：分享背景与目标
- **主题**：基于 Gemini 3 和 CLI（命令行）工具搭建一个能够自主运行的 AI 开发团队，比喻为 7x24 不眠不休的 AI 团队。
- **背景**：AI 编程发展约两年，使用 Google 的 Gemini 等工具，但人类成为效率瓶颈。目标是通过借鉴传统研发流程，用 Gemini 3 作为 AI 项目经理，减轻人类负担。

## 二、现阶段 AI 编程的痛点
- **项目规模限制**：在代码规模小（几千行到几万行）时，AI 编程体验较好；但超过十万行后，若结构不佳，AI 难以自行维护。
  - 例子：AI 生成几千行代码，但持续添加功能后逐渐失效。
- **具体问题**：
  1. **上下文丢失**：对话长度增加，AI 易忘记早期架构约定或任务，导致“胡说八道”。
  2. **幻觉与退化**：AI 可能编写不存在的 API 或跨平台错误（如安卓项目用 iOS API），或“偷懒”写虚假代码。
  3. **人类成为瓶颈**：AI 能快速写代码（如 5 分钟写 2000 行），但人类审查和测试耗时（可能 1 小时以上），开发需人类参与，停止则项目停滞。
- **总结**：AI 编程当前在战术层面高效，但缺乏战略规划，易失控。

## 三、传统研发流程与 AI 编程对比
### 3.1 传统研发流程
- **典型步骤**：
  1. 需求分析：产品经理画原型、细化需求、评审优先级。
  2. 技术评审：开发者设计技术方案，评审可行性。
  3. 编码与本地测试：开发者写代码并本地测试。
  4. CICD 与自动化测试：机器跑测试。
  5. 代码审查（CR）：经验开发者审查代码质量。
  6. QA 测试：测试人员验证功能。
  7. 产品经理验收：确认需求符合。
  8. 发布：如灰度发布。
- **特点**：流程慢、成本高，但稳定可靠，由专业人员逐层把关。

### 3.2 当前 AI 编程流程
- **典型步骤**：用户输入 Prompt → AI 生成代码 → 人类审查和测试 → 循环修改。
- **问题**：
  - 依赖 Prompt 质量，受模型训练数据影响（如 Gemini 在安卓领域更强）。
  - 随机性、幻觉导致代码不可控。
  - 人类陷入审查-修改循环，效率低下。
- **总结**：缺乏标准化流程，AI 成为“最不可控的破坏者”。

## 四、解决方案：用 Gemini 3 作为 AI 项目经理
- **核心思路**：借鉴传统流程，引入 AI 管理者（Manager）来统筹多 Agent 研发团队，实现流程规范化和自动化。
- **选择 Gemini 3 的原因**：
  - 综合能力强、多模态支持（可处理需求文档、架构图、UI 图）。
  - 上下文窗口大，能统筹整体研发计划。
- **Gemini 3 的角色**：负责监管、调度和兜底，不直接写代码，但管理任务分配、审查和异常介入。
  - 监管整个流程。
  - 动态调度：为 CLI 工具赋予不同角色和技能。
  - 异常介入：识别 CLI 工具的死循环等问题，尝试解决；解决不了则通知人类。

## 五、CLI 工具的角色分配与实操步骤
### 5.1 角色分配
- **多 Agent 系统**：开多个进程/窗口，每个 CLI 工具作为不同角色的 Agent，配备相应技能。
  - **产品经理 Agent**：细化需求，生成需求文档。
  - **架构师 Agent**：设计技术方案和架构。
  - **开发者 Agent**：如前端、后端（安卓、Go、Python等）开发者，负责编码。
  - **QA Agent**：负责测试。
  - **代码审查 Agent**：先用 AI 进行初步审查。

### 5.2 实操工作流程（完整步骤）
1. **人类输入需求**：例如“在登录页面加一个微信登录”。
2. **Gemini 3（Manager）分配任务**：将需求分配给产品经理 Agent。
3. **产品经理 Agent 执行**：查询现有文档（如历史需求），完善产品需求文档。
4. **需求传递给架构师 Agent**：进行技术设计，产出架构方案。
5. **Gemini 3 进行任务规划**：将复杂任务拆分为子任务，分配给对应 Agent（如前端和后端 Agent）。
6. **开发 Agent 执行**：
   - 前端 Agent：基于前端技术栈编码。
   - 后端 Agent：基于后端技术栈编码。
   - 避免同一需求并行修改同一文件，按顺序执行。
7. **异常处理**：
   - CLI 工具遇到问题时，先尝试自行解决。
   - 解决不了则由 Gemini 3 介入提供修复建议。
   - 若 Gemini 3 无法解决，则通知人类介入。
8. **测试阶段**：QA Agent 生成测试指令，执行自动化测试。
9. **总结与验收**：Gemini 3 生成日报总结，人类查看结果。

## 六、实战案例：架构与测试自动化
### 6.1 系统架构
- **前端**：展示 UI，显示需求阶段、执行日志。
- **引擎**：核心部分，包含数据库存储 Agent 执行数据、CLI 管理器、Manager AI（Gemini 3）。
  - CLI 封装：支持 Gemini CLI、Cursor 等工具，可自定义模型。
  - 调用循环：CLI 调用引擎基础能力，反馈到执行者。
- **流程简化图**：人类输入 → Gemini 3 管理 → 任务分配 → 开发测试 → 异常处理 → 日报总结。

### 6.2 测试自动化探索（安卓领域）
- **工具**：自研自动化测试工具。
- **流程**：
  1. AI 生成测试指令（基于需求）。
  2. 决策层（基于模型）生成指令。
  3. 执行测试：使用 OCR（文字识别）和 OpenCV（图标识别）识别屏幕元素。
  4. 结构化数据反馈给 AI，AI 决定下一步操作（如点击、滑动、输入）。
- **扩展性**：基于视觉，可扩展到桌面端。

## 七、成本和展望
- **Token 成本问题**：使用顶尖模型（如国外）成本高，一天烧几百美金常见。
- **成本下降预测**：业界共识，Token 成本每 10-12 个月下降 10 倍。
- **未来展望**：个人化软件工厂时代，只要用户有想法，AI 就能帮忙实现，软件开发变得平常。

## 八、回答常见问题
1. **架构师如何用 AI 改善工作？**
   - 梳理系统架构形成结构化文档，让 AI 提供建议和灵感；但 AI 在特定领域不会比人类强太多。
2. **如何避免子任务信息丢失和偏差？**
   - 避免同一需求并行修改同一文件；按顺序执行；Agent 读文档和代码；Gemini 3 调控和回退。
3. **7x24 运行的容错机制？**
   - 日志和工作流状态化，从停止点重启；出错用版本管理回退；人类定期检查（如每小时一次），复杂场景需人类介入。

## 核心要点总结
本次分享探讨了当前 AI 编程的痛点，如上下文丢失和人类瓶颈，提出借鉴传统研发流程的解决方案：使用 Gemini 3 作为 AI 项目经理，统筹多个 CLI 工具作为不同角色的 Agent，实现任务分配、审查和异常处理的自动化，从而构建一个 7x24 运行的 AI 开发团队。通过实战案例和测试自动化探索，展示了该架构的可行性，并指出未来随着 Token 成本下降，个人化软件工厂将成为可能，让软件开发更加普及和高效。
