# 一个CLI干掉所有MCP工具，省99%的token mcp2cli

> [探索未至之境](https://space.bilibili.com/441831884) | [BV1c8NFzhEMi](https://www.bilibili.com/video/BV1c8NFzhEMi) | 时长 3:33

## 笔记正文

### 1. 痛点分析
- **问题根源**：MCP协议设计要求每轮对话客户端传递所有工具的完整schema，导致大量Token浪费。
- **具体数据**：
  - 每个工具的schema约占用120个Token。
  - 示例：30个工具，15轮对话，浪费54000个Token。
  - 极端案例：120个工具，25轮对话，光是schema就浪费36万Token。
- **核心影响**：这不是技术缺陷，而是成本问题（烧钱）。

### 2. 解决方案：MCP2CLI的懒发现模式
- **核心思路**：从一次性灌入所有工具描述转变为AI按需查询，动态获取工具信息。
- **操作步骤**：
  1. **执行List命令**：仅返回工具名称列表，每个工具约16个Token（极简）。
  2. **执行Help命令**：AI觉得某个工具有用时，查询其详细参数，约80-200个Token。
  3. **确认后执行**：确认需要后，才真正调用工具。
- **节省效果**：
  - 30个工具，15轮对话：Token消耗从54000降至2300，节省96%。
  - 120个工具：节省高达99%。

### 3. 实现细节
- **项目结构**：核心代码仅一个Python文件，依赖极少（HTTPX、MCP、PyML三个库）。
- **支持模式**：
  - **MCP-HTTP**：连接远程服务器。
  - **MCP-Steel**：启动本地进程。
  - **OpenAPI**：直接解析API规范。
- **统一处理**：三种来源均转化为内部模型，动态生成命令行界面（CLI）。
- **内置功能**：
  - OAuth 2.0认证和API密钥管理。
  - 响应缓存机制。
  - TUN紧凑输出格式：专为节省Token设计的专用格式。

### 4. 社区反馈与争议
- **项目热度**：发布当天获得372颗星，登上Hacker News首页，评论区热议。
- **支持观点**：
  - CLI通用、简单，适配任何大模型。
- **反对质疑**：
  - 懒发现模式（先List再Help）可能降低工具调用的准确率。
  - 多一步查询往返是否增加延迟？
  - 作者仅展示Token节省数据，未提供准确率对比。
- **行业背景**：社区发现约100个类似项目，说明CLI取代MCP的思路并非独创，存在路线之争（CLI vs MCP）。

### 5. 总结
- **核心成就**：MCP2CLI以最少代码（单文件）解决了MCP最大的痛点——Token浪费，实现99%的节省。
- **优势总结**：支持三种介入模式，内置认证、缓存等功能，简洁高效。
- **待验证问题**：准确率是否受影响以及延迟增加情况，需更多实测数据支撑。

核心要点：MCP2CLI通过懒发现模式（按需查询工具描述）大幅降低了MCP工具调用中的Token消耗，实现高达99%的节省，项目实现极简且功能全面，但社区对其准确率和性能影响存在争议，后续需进一步实测验证。
