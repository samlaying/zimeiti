## 多Agent系统设计笔记：最稳健的拆分起点——验证型子Agent

### 一、核心问题与回顾
*   **背景**：本视频是《Building Multi-agent Systems: When and How to Use Them》系列的第三期，前两期已讲解何时采用多agent架构及如何划分agent边界。
*   **核心问题**：在决定采用多agent架构后，最安全、最不易出错的第一刀（即第一个拆分出去的子agent）应该切在哪里？
*   **核心答案**：最稳健的拆分起点是**验证型子Agent**，而非更复杂的思考型或写作型agent。

### 二、为什么验证型子Agent最稳健？
#### 1. 典型角色模式
*   **主Agent**：负责核心任务（如写代码、生成内容），拥有完整的上下文和理解能力。
*   **验证Agent**：完全不关心任务执行过程，不理解结果如何产生。其唯一职责是回答一个问题：“**结果是否满足预先定义的标准？**”

#### 2. 关键逻辑澄清：与“拆分测试”的本质区别
*   **常见疑问**：这与上一期反对“拆分编写测试”的“传声筒架构”是否矛盾？
*   **答案与精髓**：
    *   **反对的是**：拆分**编写测试**。这是一个**白盒任务**，需要理解业务逻辑、代码结构和实现意图，离不开大量上下文。强行拆分会导致子agent成为“瞎子”，陷入需要不断提问的“传声筒陷阱”。
    *   **推荐的是**：拆分**执行验证**。这是一个**黑盒任务**，验证agent无需知道代码如何实现，只需将产物（如代码、文档）与预设标准进行**机械式对照**。
*   **生动比喻**：
    *   写测试如同**学生写作文**，必须自己构思，不能代笔（上下文内聚）。
    *   执行验证如同**老师判卷子**，不在乎写作心理活动，只依据字数、是否跑题等客观标准评分（标准客观独立）。

#### 3. 核心优势
*   **天然符合“以上下文为中心”的分解原则**：验证任务是上下文最“干净”的一类工作，非常适合拆分。

### 三、验证型Agent的广泛应用场景
验证Agent不仅是代码测试工具，任何**标准明确、无需理解过程**的工作均可交给它，是系统中成本最低但收益最高的“守门员”。
*   **代码测试**：运行测试套件并报告所有失败项。
*   **合规检查**：主agent写合同，验证agent拿着法务手册核对条款是否存在，无需懂法律逻辑。
*   **事实核查**：主agent写文章，验证agent专门去搜索引擎验证每一个数据的真实性。
*   **格式校验**：专门检查输出的JSON字段是否符合规范。

### 四、实战陷阱与防偷懒铁律
#### 1. 陷阱：过早胜利问题
*   **现象**：由于LLM倾向于顺从用户以尽快结束对话，验证子Agent常有“偷懒”倾向。例如，只运行一两个最简单的测试通过后，就立刻宣布成功。这会导致系统上线后在复杂边界情况下崩溃。

#### 2. 防偷懒铁律（建议写入System Prompt）
1.  **设定具体标准**：不要说“确保能运行”，要说“**运行完整的测试套件并报告所有的失败项**”。
2.  **强制全面检查**：要求**覆盖边界情况**（如空值、超长文本）。
3.  **进行反向测试**：指令验证者**试着输入一些会导致失败的数据**，并确认系统真的报错了。如果无论输入什么都“成功”，则说明测试是假的。
4.  **使用强势指令**：使用强起始句，例如“**你必须运行完所有测试才能标记为通过**”。

### 五、落地检查清单
在动手编写多agent系统代码前，请再次问自己三个问题：
1.  **真的有物理限制吗？** 单体能解决的问题，坚持使用单体。
2.  **是按上下文拆分的吗？** 拆分边界是基于真实的知识需求，还是盲目模仿流水线？
3.  **有清晰的验证点吗？** 你能否定义出明确的标准，让子agent进行黑盒验证？

### 总结
整个系列的核心思想是：**从单体开始，只在真实约束出现时引入多agent。而在引入时，最稳的第一个拆分点不是更复杂的思考agent，而是上下文最干净的验证型子Agent。** 通过为其设定具体、全面的验证标准并防止其“偷懒”，可以构建出既安全又高效的多agent系统。
