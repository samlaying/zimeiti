# AI生成内容不稳定？用这套“多智能体审查系统”彻底解决

## 一、核心问题：AI生成内容为何不稳定？
AI（如Claude）本身能力很强，但生成复杂内容时质量不稳定，主要原因并非其“不聪明”，而是**缺乏有效的、结构化的审查机制**。常见问题有四点：
1.  **跨会话上下文丢失**：不同会话中，AI无法记住之前设定的规则（如将β定义为回归系数）。
2.  **质量参差不齐**：处理长文档时注意力会分散（模型的通病），导致前后输出质量不一致（如前40页排版完美，第40页后文本溢出）。
3.  **审查全靠人工肉眼**：人工检查如140张幻灯片，效率低且容易遗漏。
4.  **专业内容无人校验**：语法检查工具（如Grammarly）无法校验数学公式下标、代码逻辑、教学流程等专业内容的正确性（例如，分解定理的正负号错误、蒙特卡洛模拟次数在幻灯片与代码中不一致）。

### 为什么不让一个AI包办所有审查？
让单个AI同时负责语法、布局、数学、代码、教学法等多个维度，其“注意力”会被严重稀释，导致每个维度都只能做到约70分，大量细节错误被遗漏。
**解决方案是专业分工**：让多个专业AI智能体（Agent）各司其职，将100%的注意力集中在一个维度上，进行深度审查。深度审查远比广度扫描更可靠。

**专业分工审查示例表：**
| 智能体角色 | 负责维度 | 能发现的问题示例 |
| :--- | :--- | :--- |
| 教校员 | 语法、拼写 | `principal` 和 `principle` 混用 |
| 幻灯片审计员 | 布局、排版 | 第37张幻灯片文本超出边界 |
| 教学法审查员 | 教学流程、逻辑 | 定理3.1之前缺乏引导语 |
| R代码审查员 | 代码质量、可复现性 | 脚本中未设置`set.seed()`；R包在变量缺失时静默丢弃观测值 |
| 领域审查员 | 专业交叉验证 | 幻灯片称“跑了1万次模拟”，但R脚本中是“1000次” |
| Quarto评审者 | 格式转换一致性 | Beamer原文件有完整公式，Quarto输出仍是省略号占位符 |

## 二、解决方案：多智能体审查系统整体架构
该系统是一个工程化的工作流，而非单一提示词技巧。其配置像一个项目的“操作系统”，共五层：

### 1. `CLAUDE.md`（项目宪法）
- 每次新会话自动加载。
- **必须精简**（约120行），只写核心规则、文件夹结构、关键命令速查表。
- **原因**：AI能可靠遵循的自定义指令总数约150条（系统提示已占用~50条）。文件过长会导致后续规则被忽略（注意力预算用完）。

### 2. `Rules`（路径作用规则）
- 存放在 `.claude/rules` 目录下。
- **关键设计**：每条规则通过 `Path` 字段指定生效路径（如只适用于`.R`文件或`.tex`文件）。
- 实现“按需加载”，不编写相关文件时不占用注意力预算。模板中共有14条规则。

### 3. `Skills`（常用技能/命令）
- 19个编写的快捷命令，如 `Compile LaTeX`、`Proofread`、`Deploy`。

### 4. `Agents`（10个专业智能体）
- 系统的核心战斗力，各司其职。

### 5. `Settings` & `Hooks`（设置与钩子）
- 控制权限和自动化行为。

**核心设计原则——注意力预算**：
- 不要把所有规则堆砌在`CLAUDE.md`中。
- 将“项目宪法”级内容放入`CLAUDE.md`。
- 将详细标准、规范放入**路径作用的Rules**中，按需加载，节省注意力预算。

## 三、四大核心机制详解
### 1. 对抗式QA循环（最强设计模式）
- **精髓：角色分离与制衡**。
    - **评审者（Critic）**：只有**只读权限**，不能修改文件。其KPI是**找出问题**，报告不会含糊。
    - **修复者（Fixer）**：有读写权限，负责修复。
- **流程**：Critic评审 → Fixer修复 → **Critic复审**（检查修复是否引入新问题或遗漏）。此循环最多进行5轮，直至质量达标。
- **为何有效**：打破了AI“对自己工作说‘看起来不错’”的倾向。角色分离消除了自我评价偏差。
- **真实案例**：在将Beamer幻灯片翻译成Quarto时，初审发现核心公式被遗漏为省略号；修复后复审，又发现8处结果模型缺少一个X参数。经4轮循环，两个版本最终完全一致。

### 2. 编排器（总承包商）
- 用户只需用自然语言下达任务（如“把第五讲翻译成Quarto”）。
- 编排器自动接管全流程：
    1.  **规划（关键前置步骤）**：分析任务，设计方案，列出步骤，**并将计划保存到磁盘文件**（而非仅存于对话上下文，以防压缩丢失）。
    2.  **实施**：按计划创建文件，翻译内容。
    3.  **验证**：直接渲染（如Quarto渲染为HTML）检查输出。
    4.  **审查**：根据修改的文件类型，**自动选择并派发合适的智能体并行审查**。
    5.  **应用修复**：严重问题优先。
    6.  **重新验证**：确认修复未引入新问题。
    7.  **打分**：启动质量门禁。
- **上下文恢复**：若中途丢失上下文，只需说“读一下Plans目录里的那个计划文件”即可快速恢复。

### 3. 质量门禁
- 每份输出获得0-100分的质量评分。
- **硬性标准**：**低于80分，不允许提交**。
- **评分体系（按问题严重性扣分）**：
    - 公式错误/截断：-20分
    - 引用损坏：-15分
    - 数学公式错误：-10分
    - 文本溢出：-5分
    - 标签重叠：-5分
    - 标记法不一致：-3分
- **作用**：该评分是编排器循环的**终止条件**。分数不达标，则继续审查-修复循环。

### 4. 四层记忆系统
解决跨会话上下文丢失问题，确保重要信息持久化。
1.  **`CLAUDE.md`**：项目宪法，每次会话加载。
2.  **`MEMORY.md`**：存储“修正记忆”（Learn标签），如`Learn: CitationPostLasso是Beloni 2013年的论文`。跨会话可查，避免重复犯错。
3.  **`Plans`目录**：保存每个非琐碎任务的执行计划（磁盘文件，抗压缩）。
4.  **`Session Logs`**：记录决策推理（**为什么**这样做），而不仅仅是操作日志（**做了什么**）。
- **设计核心**：前四层全部写在磁盘上。只有第五层“对话上下文”是短暂的，会被自动压缩。确保所有重要信息在压缩前已安全落盘。

### 5. Hooks（钩子）
- 定义在`settings.json`中，**不受上下文状态影响**，每次都会触发。
- 用于快速、机械性的检查，**不要在其中调用LLM**（以免累积延迟）。
- **示例钩子**：
    - **会话日志提醒**：检查自上次更新日志以来的轮次，超阈值则提醒。
    - **桌面通知**：当AI等待输入或需要权限批准时，弹出系统提醒。
    - **文件保护**：阻止AI意外编辑参考文献等敏感文件。
    - **上下文快照**：在自动压缩触发前，保存重要推理笔记到会话日志。

## 四、三大实战应用场景
### 场景一：将Beamer幻灯片翻译为Quarto网页版
- **用户指令**：`把第五讲翻译成Quarto`
- **编排器自动完成**：
    - 创建QMD文件，逐页翻译。
    - 将`TikZ`图表转为SVG矢量图，`ggplot`转为交互式`Plotly`。
    - 启动对抗式QA，评审者逐页对比原文件与输出。
    - 修复所有不一致。
    - 质量评分92分，通过门禁。
    - 自动渲染为HTML，部署到GitHub Pages。

### 场景二：从零创建一个新讲座（`Create Lecture`命令）
1.  **收集原材料**：论文、大纲、参考文献。
2.  **设计幻灯片结构**：规划概念依赖关系。
3.  **起草Beamer幻灯片**。
4.  **生成配套图表**（如蒙特卡洛模拟可视化）。
5.  **核心审查**：最多6个智能体并行审查（视觉、教学法、语法、图表、内容、领域）。
6.  **翻译为Quarto并部署**。

### 场景三：复现优先的编码协议（适合论文复现）
1.  **清点“黄金标准”数值**：从原始论文表格中精确记录关键数值。
2.  **精确翻译代码**：匹配原始规格（变量、聚类、样本筛选）。
3.  **逐个验证匹配**：估计值容差<0.01，标准误容差<0.05。**不匹配则强制停止**，直至查明原因。
4.  **验证全部通过后**，才允许进行扩展。
- **价值**：曾靠此协议发现一个被广泛使用的R包因细微规格问题默默产生错误估计值。

## 五、适用项目与渐进式采用
### 什么项目适合使用？
- **适合**：学术课件、技术文档、代码+文档混合项目、多格式输出、需要质量保证的团队项目。
- **不适合**：一次性简单问答、快速原型、纯文本内容、短期小项目（文件少于10个）。

### 如何渐进式采用？
1.  **第一阶段（当天）**：Fork仓库，填好`CLAUDE.md`，使用2-3个基础`Skills`（如编译、校对、提交）。
2.  **第一周**：遇到具体问题（如R脚本不规范），添加3-5条**路径作用规则**。
3.  **第二周**：熟悉后，启用**编排器**和**对抗式QA**。
4.  **第一个月**：根据领域，定制**领域审查员**。
5.  **持续积累**：用`Learn`标签记录每个修正，用`Session Logs`记录决策原因。

## 六、六条实战经验总结
1.  **遵守注意力预算**：`CLAUDE.md`保持150行以内，详细规则放入路径作用的Rules。
2.  **信任对抗模式**：让它严格跑完，能发现你永远手动找不到的问题。
3.  **验证一切**：每次修改后都编译/渲染，不要跳过。
4.  **用`Learn`标签记录修正**：跨会话持久化，避免重复犯错。
5.  **用`Session Logs`记录决策原因**：未来的你会感谢现在的自己。
6.  **尽早使用“魔鬼代言人”**：在基于脆弱结构写太多内容前，先让AI用具体问题挑战你的设计。

## 总结
本视频介绍了一套完整的“多智能体审查系统”工程化工作流，旨在彻底解决AI生成内容不稳定的问题。其核心在于**专业分工**（10个智能体各负责一个维度）、**对抗式QA循环**（评审者与修复者分离制衡）、**编排器自动协调**、**严格的质量门禁**（80分红线）、以及**四层持久化记忆系统**。该系统强调**渐进式采用**，从一个简单的`CLAUDE.md`和基础技能开始，按需扩展规则和智能体，最终构建一个能自我审查、自我修复、持续积累经验的AI协作系统，确保复杂项目（如学术课件、代码复现）的输出质量达到专业标准。
