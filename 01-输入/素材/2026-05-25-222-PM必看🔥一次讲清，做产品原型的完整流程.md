# 产品原型制作完整流程笔记

## 一、画原型前
### 1. 需求调研
- **业务调研**：梳理业务流程，了解业务需求。
- **用户调研**：生成需求调研报告，收集用户需求。
- **建议**：使用AI Skill整理用户故事，提高效率；参考笔记中的提示词示例（如如何编写用户故事）。

### 2. 需求整理
- **用户故事**：编写用户故事，可借助AI Skill快速生成。
- **示例**：笔记中提供了编写用户故事的提示词，方便直接使用。

### 3. 竞品分析
- **目的**：明确分析目标，指导后续工作。
- **步骤**：
  1. 行业调研：了解市场环境。
  2. 确定竞品：选择相关竞品。
  3. 数据对比分析：收集并对比数据。
  4. 数据结论：总结分析结果。
- **维度**：
  - 业务层面：在产品概念阶段关注，如商业模式。
  - 产品层面：在概念阶段关注，如产品定位。
  - 功能层面：在产品规划和开发阶段关注，包括功能点、交互布局、视觉体验。
- **重点**：对于产品原型，功能层面是核心调研点。

### 4. 梳理产品结构图
- **定义**：参考笔记介绍产品结构图的组成。
- **示例**：B端管理页面示例，结构包括搜索条件、按钮、左侧树、列表、页面操作等。
- **2C应用**：类似结构整理，适用于消费者端产品。
- **输出**：生成产品结构图，作为后续设计基础。

## 二、画原型中
### 1. 两种方案
#### (1) 精准控制
- **描述**：将脑中的想法精确落地，确保逻辑清晰。
- **步骤**：
  1. 整理产品说明书：基于产品结构图。
  2. 转为详细功能描述：参考笔记中如何将产品结构图转为详细描述。
  3. 生成高保真产品原型：使用工具实现精确设计。
- **适用**：适合逻辑复杂、需要严格控制的需求。

#### (2) 边做边探索
- **描述**：将模糊想法通过迭代逐步完善。
- **适用**：
  - 简单需求：输出初步想法，使用大模型直接迭代优化。
  - 复杂需求：先打磨最小可行产品（MVP），再迭代；或逆向总结产品原型生成PRD文档，理清逻辑。
- **逆向总结示例**：将产品原型转成PRD文档，便于理解大模型生成的复杂信息（参考笔记）。

### 2. 工具选择
- **原型工具**：Figma、MakeKaliCat等，可生成可编辑设计稿。
- **应用开发工具**：如LavaBall，生成前端页面。
- **AI编程工具**：Tree、Cursor、CutureX等，可生成原型代码。
- **大模型**：Gemini、Dipsic、Kimi等，辅助生成内容。
- **建议**：按需选择工具，确保工具支持编辑和迭代。

### 3. 工作流总结
- 不管哪种方案，最终都可生成产品说明书来指导原型生成。
- 对于自用小工具，逻辑简单时可跳过说明书，直接根据核心想法生成。

## 三、画原型后
### 1. 代码管理
- **工具**：使用GitHub或GitLab管理代码，替代传统原型工具（如Axure）的云端管理。
- **对比**：WebCoding方式更注重代码版本控制。

### 2. 云端部署
- **方法**：使用云端部署工具或公司内部服务器发布原型。
- **示例**：笔记介绍了如何将AI生成的原型发布到云端，方便团队访问。

### 3. 标注
- **挑战**：WebCoding方式下，标注交互逻辑和说明较麻烦。
- **解决方案**：开发浏览器插件，实现对产品交互逻辑和说明的标注。

### 4. 产品评审
- **现场修改**：使用WebCoding工具（如LavaBall）可在评审中快速修改原型，支持实时调整。
- **会议纪要**：使用豆包、飞树、丁钉等工具录制会议，自动生成纪要；可制作Skill按公司模板输出。

### 5. PRD文档生成
- **内容**：参考笔记介绍AI产品PRD文档的必备内容。
- **Skill示例**：
  - 根据产品创意或描述生成PRD。
  - 根据原型逆向拆解生成PRD（基于“边做边探索”方案）。

### 6. 研发跟进
- **部署与标注**：研发通过云端原型查看，并借助标注工具理解设计。
- **持续迭代**：原型需根据研发反馈修正，记录变更日志。
- **拉通会议**：每日站会同步进度，周会重点拉通变更情况，确保团队共识。

## 四、总结
核心要点：产品原型制作是一个系统化流程，涵盖画原型前的需求调研、竞品分析和产品结构图梳理；画原型中的精准控制或边做边探索方式，结合AI工具生成原型；以及画原型后的代码管理、云端部署、标注、评审、PRD生成和研发跟进。整个过程强调迭代优化、文档化和团队协作，通过AI辅助提升效率，确保产品设计可控且持续改进。
