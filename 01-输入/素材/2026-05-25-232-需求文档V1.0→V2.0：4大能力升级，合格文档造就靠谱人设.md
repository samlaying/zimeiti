# 需求文档V1.0→V2.0：4大能力升级，合格文档造就靠谱人设

> [秀明说](https://space.bilibili.com/3546590214097572) | [BV1jbFTeTEgR](https://www.bilibili.com/video/BV1jbFTeTEgR) | 时长 19:48

## 需求文档V1.0→V2.0升级指南
### 1. V2.0版本的四大能力升级
- **更照顾阅读者**：通过格式排版、标记和用心写文档提升可读性。没有技巧，全靠用心，以始为终地写好文档。
- **逻辑自洽基础上的兼容意识**：增加时序图表达不同部门和模块之间的合作，强调接口和系统交互。
- **交互细节明确**：详细描述交互元素，如按钮、文本框等，让前端工程师能清楚理解。
- **深层数据表字段认知**：在文档中注明常用字段，如UID，体现对数据层面的理解。

### 2. 名词解释的重要性
- **定义关键术语**：在文档开头添加名词解释，确保团队理解一致，尤其适用于新增项目或模块。
- **例子**：
  - UID：用户注册账号的唯一ID，是用户表的索引。
  - Toast：一个术语示例，用于解释可能引起混淆的词汇。
- **建议**：假设别人可能不理解，尤其在新项目中，明确定义所有关键术语。

### 3. 格式和符号的优化
- **括号表示按钮**：用括号将按钮类元素括起来，如（注册新账号），直观显示交互点。
- **符号表示条件**：用特殊符号（如竖线）表示条件中的多个条件，使逻辑清晰。
- **横线和斜体**：用于表示错误消息或重要信息，如验证码发送。
- **表格化交互描述**：将交互元素列成表格，包括字段是否必填、样式、逻辑、报错等，增强结构化。

### 4. 交互部分的详细描述
- **核心要素**：必填性校验、样式、逻辑、报错处理。
- **例子：手机号和验证码**：
  - 必填性：均为必填，需校验。
  - 样式：文本框，Placeholder为灰色提示文字。
  - 逻辑：默认为空，逻辑简单。
  - 报错：手机号需为中国大陆11位纯数字，开头为1；验证码校验与手机号的关联关系，如错误则提示“验证码错误”。
- **其他元素**：如下拉框（dropdown）用于区号选择，免去输入校验。
- **目的**：通过表格化方式，强制逻辑严谨，避免遗漏交互细节。

### 5. 时序图描述系统交互
- **应用场景**：B端产品中描述系统间交互，如用户、账号中心、短信服务商。
- **流程步骤**（对应时序图序号）：
  1. 用户点击发送验证码。
  2. 账号中心校验手机号有效性（后端服务兜底）。
  3. 生成六位纯数字验证码，并绑定手机号与验证码（关系保留10分钟，相同手机号重复发不生成新验证码）。
  4. 组装短信内容（引用用力1.5），调用短信服务商接口（如Send SMS，入参包括手机号、验证码，出参为成功或失败）。
  5. 返回发送结果给用户。
  6. 用户填写验证码并提交注册。
  7. 后台校验手机号有效性、验证码关联关系（在10分钟内有效）。
  8. 创建新账号（包括UID、手机号、默认密码等，UID使用随机算法生成）。
  9. 注册成功后跳转到首页。
- **建议**：文档与图像有映射关系，文字描述对应时序图序号，确保清晰。

### 6. 接口描述和文档引用
- **接口描述**：详细说明入参和出参，如Send SMS接口的字段名、含义和数据类型。
- **引用关系**：文档间引用，如短信内容引用用力1.5，便于后期维护，只改一处即可。

### 7. 实操步骤和建议
- **写文档的步骤**：
  1. 以始为终，用心写文档，照顾阅读者。
  2. 添加名词解释，定义所有关键术语。
  3. 结构化描述功能和交互，使用表格、符号等增强可读性。
  4. 用时序图描述系统间交互，确保逻辑自洽。
  5. 深入数据表字段，在文档中注明常用字段。
- **V1.0到V2.0改进**：V1.0文档难读、交互模糊；V2.0通过格式优化、时序图、详细交互描述和字段认知，使文档一次通过。

## 总结核心要点
需求文档从V1.0升级到V2.0，核心在于提升文档的可读性、逻辑严谨性和深度。通过用心照顾阅读者、添加名词解释、优化格式符号、详细描述交互、使用时序图系统化表达接口关系，并深入数据表字段认知，使文档成为合格、靠谱的工具，促进团队协作和项目一次性通过。
