> [五里墩茶社](https://space.bilibili.com/615957867) | [BV1Wc9KBsEQf](https://www.bilibili.com/video/BV1Wc9KBsEQf) | 时长 12:46

## 编程智能体架构对比：OpenAI Codex CLI vs Anthropic Claude Code
本文基于视频内容，从技术选型、核心循环、工具系统、安全机制等多个维度，深度剖析OpenAI Codex CLI和Anthropic Claude Code这两个代表性AI编程智能体的设计与实现差异。

### 一、 核心设计哲学
*   **OpenAI Codex CLI (Codex)**：**“沙箱优先” (Containment First)**。核心思想是利用操作系统内核级沙箱来限制AI能做的事情。模型可以在沙箱内自由执行任何命令，但由操作系统内核挡住危险操作。
*   **Anthropic Claude Code (Claude Code)**：**“分析与审批优先”**。核心思想是执行前分析一切，有风险就询问用户。它没有内核沙箱，而是依赖一个**大模型分类器**（即额外调用一次Claude API）来判断每条命令的风险。安全的命令自动执行，危险的弹窗询问，特别危险的直接拒绝。

### 二、 技术选型与项目结构
#### **Codex CLI**
*   **语言**：Rust。
*   **结构**：一个包含 **81个Crate** 的庞大代码仓库（monorepo）。模块划分清晰（如CLI、TUI、核心组件、沙箱环境、执行策略等），编译器强制执行模块边界。
*   **分发**：最终编译为一个**静态链接的原生二进制文件**。通过NPM分发，但NPM包只是一个“包装器”（wrapper），用于检测用户平台并启动对应的Rust二进制文件。
*   **API**：使用 **OpenAI Responses API**，支持 **SSE 和 WebSocket**。WebSocket 有连接池和预热机制。

#### **Claude Code**
*   **语言**：TypeScript。
*   **结构**：单体包结构。拥有一个自定义的、基于**React**和**React Ink**的终端渲染器（并非NPM上的Ink，而是Fork出的一套）。
*   **分发**：现在推荐的安装方式是**原生二进制，支持自动更新**。
*   **API**：使用 **Anthropic Messages API**，仅支持 **SSE**。但拥有完善的**prompt缓存机制**，将系统提示词分为可缓存的静态部分和每轮更新的动态部分，以节省成本。

### 三、 Agent Loop (智能体循环)
两者本质是同一模式：接收输入 -> 调用模型 -> 执行工具 -> 将结果反馈给模型 -> 循环。但实现方式不同。
#### **Codex CLI：基于“轮次”(Turn)的状态机**
*   每轮执行都有一个明确的 **Turn Context**（轮次上下文），包含模型信息、沙箱策略、权限配置。
*   **基本流程**：构建提示词 -> 调用API（流式返回）-> 提取工具调用 -> 执行 -> 更新历史 -> 决定是否继续。
*   **没有最大轮次限制**：只要有工具在执行，就会继续。
*   **流式处理与结果同步**：在API流式返回期间就开始执行工具调用。它将每个工具调用派生为一个独立任务放进容器，使用**读写锁**控制并发（读取工具并行，写入工具独占）。**关键点：所有工具执行结果在流结束时统一发出**，即轮次的边界是同步点。

#### **Claude Code：基于Generator的异步流式循环**
*   核心是一个**异步Generator函数**，通过`Yield`不断产出事件。循环更流畅，没有硬性的轮次边界。
*   **流式处理与结果同步**：同样在流式传输期间就开始执行工具。使用 **Streaming to Executor 状态机**，工具一从流中解析出来就立刻执行（读取工具最多10个并行，写入工具串行）。**关键点：它可以在其他工具还在执行时，通过`yield`发出已完成的结果**，同步粒度更细。

### 四、 工具系统设计
#### **Codex CLI：少而精，哲学是“给模型一个Shell”**
*   **工具数量**：约 **15个** handler。
*   **核心工具**：`shell`（用于执行任意命令）和 `apply_patch`（用于编辑文件）。
*   **哲学**：模型知道怎么用`shell`。想搜索文件？模型自己跑`rg`。想看目录结构？跑`ls`。所有操作都通过`shell`进入内核沙箱。
*   **文件搜索**：**不自带搜索工具**。系统提示词指导模型优先使用`rg`(ripgrep)。如果用户机器上没有，模型会收到错误并自动降级到`grep`或`find`。
*   **文件编辑**：使用自研的 **`apply_patch`** 工具（纯Rust实现）。这不是标准的unified diff，它有一个**4G模糊匹配算法**来定位代码位置（精确匹配、忽略尾部空白、忽略首尾空白、Unicode标点归一化）。支持一次调用**同时修改、创建、删除多个文件**，且是**事务性**的。

#### **Claude Code：多而专，生态丰富**
*   **工具数量**：目测有 **40多个** 专用工具。
*   **工具类型**：涵盖文件搜索(`glob`, `grep`)、文件编辑(`file_edit`)、网络(`web_fetch`, `web_search`)、LSP、子智能体(`agent`)等。
*   **设计特点**：每个工具都有自己的输入校验、权限检查和输出格式化。系统提示词中甚至会明确指示（如“永远不要用Bash跑`grep`，要用`grep`工具”）。
*   **文件搜索**：**自带ripgrep**。它将`rg`二进制文件打包进应用程序（编译进主二进制或作为按平台分发的vendor二进制）。`grep`工具是对`rg`的完整封装。
*   **文件编辑**：使用自研的 **`file_edit`** 工具（纯TypeScript实现）。底层是JavaScript的`String.replace`和`replaceAll`，但增加了**智能的引号归一化**（如单引号、直引号匹配）、编码检测和换行符保留。**一次调用只改一个文件的一个位置**，设计哲学是简单、可靠、安全。

### 五、 安全与沙箱机制 (最大架构差异)
#### **Codex CLI：内核级沙箱**
*   在macOS、Linux、Windows上使用不同的底层技术实现。
*   **提供三种模式**：
    1.  **Read-only (默认)**：只能读取，不能写入。
    2.  **Workspace write**：允许在工作目录写入。
    3.  **Full access**：完全访问（相当于关闭沙箱）。
*   **特点**：安全性强，但不够灵活，需要操作系统级别支持，各平台实现略有不同。

#### **Claude Code：应用级权限控制**
*   **没有内核沙箱**。安全靠一个**多层防御体系**叠加：
    1.  **大模型分类器**：调用Claude API判断命令风险。
    2.  **路径白名单与黑名单**。
    3.  **用户交互式审批的对话框**。
    4.  远程管理的企业级设置推送。
*   **特点**：更灵活，但理论上安全性弱于内核沙箱。如果大模型分类器“漏判”了一个恶意命令，就没有后备防线了。

### 六、 上下文压缩与长上下文管理
随着对话变长，上下文窗口会被填满，两者都需要压缩策略。
*   **Codex CLI**：策略相对简单。使用`GoSnackShot`标记对旧消息做摘要，可以调用远程API来压缩。
*   **Claude Code**：策略更复杂，有 **五层压缩策略**：上下文折叠、响应式压缩、主动压缩、剪切压缩和微压缩。此外还有一个 **Token预算机制**：当模型认为任务完成但预算未用完时，系统会注入一条“继续”的消息让它接着干；如果连续三次催促，模型每次输出都少于500个token，系统会判定智能体“空转”并提前退出。

### 七、 多智能体架构
*   **Codex CLI**：子智能体是**进程内**的，通过**共享内存**和**mailbox异步通道**通信。因此可以**fork父级的完整对话历史**。
*   **Claude Code**：子智能体是**独立进程**。每个子智能体只拿到一个任务描述，**从零开始**，没有父级历史。可以选择在独立的工作树（Git Worktree）中隔离执行。

### 核心要点总结
OpenAI Codex CLI和Anthropic Claude Code代表了两种截然不同的AI编程智能体工程哲学。**Codex CLI是“系统工程”路线**，使用Rust构建，以内核级沙箱为核心安全支柱，工具设计少而精（核心是Shell），通过硬件隔离提供强安全性，体现了类型安全、模块清晰和高性能的系统编程思维。**Claude Code是“产品与应用”路线**，使用TypeScript构建，以大模型驱动的风险分析和审批流程为核心安全支柱，提供了超过40个专用工具的丰富生态和企业级管理能力，其设计更侧重于灵活性、用户体验和产品集成。尽管安全机制和工具实现存在根本差异，但两者的核心智能体循环（基于流式API调用并执行工具）是相同的，所有外围的工程决策都是围绕这个循环进行的。选择哪一个，取决于项目对安全强度、灵活性、开发语言和工具生态的不同权衡。
