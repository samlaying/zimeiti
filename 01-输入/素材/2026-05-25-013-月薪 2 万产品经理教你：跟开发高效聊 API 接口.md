# 月薪 2 万产品经理教你：跟开发高效聊 API 接口

> [产品经理远远](https://space.bilibili.com/1242710972) | [BV1zB556CEaA](https://www.bilibili.com/video/BV1zB556CEaA) | 时长 10:58

## API接口对产品经理的重要性
*   **常见痛点**：不懂技术易被开发质疑；听不懂开发讨论的接口；因未考虑接口对接问题导致项目排期延误。
*   **必要性**：月薪一万五至两万以上的产品经理必须理解API接口。
*   **核心目标**：通过掌握API接口知识，与开发实现高效、同频的沟通。

## 系统流程图：描绘系统间协作蓝图
### 定义与作用
*   系统流程图用于展示跨系统协作时，各个子系统所承担的职责与数据交互流程。
*   在PRD文档中，它是梳理API接口清单的前提。

### 关键名词
*   **拉取**：下游系统主动向上游系统获取数据。
*   **推送**：上游系统将数据主动发送给下游系统。
*   **同步**：调用方发起请求后，实时（同步）得到结果反馈。
*   **异步**：调用方发起请求后，无法立即得到最终结果，需等待另一方完成后续处理后再通过回调通知。

### 设计规则与技巧
1.  **接口提供规则**：**箭头指向谁，谁就提供接口**。通常是核心系统（如订单系统）主动发起调用。
2.  **数据推送原则**：**谁是数据源的上游方，谁就有义务将数据推送给下游**。例如，订单系统（上游）需向WMS系统（下游）推送发货单。
3.  **例外**：也存在下游系统“拉取”数据的场景，例如平台定义标准，由多个商家自行拉取订单。

### 举例：创建待支付订单的系统流程
当用户前端点击“提交订单”时，后端的订单系统需要执行一系列跨系统调用：
1.  **调用商品系统**：校验商品上下架状态。
2.  **调用营销系统**：校验优惠券/券码状态、冻结优惠；校验秒杀/拼团活动的状态、库存及用户限购。
3.  **调用库存系统**：锁定销售层库存。
以上任一环节失败，都会导致订单创建失败。

## API接口清单：明确对接责任
### 作用与格式
*   **作用**：基于系统流程图，明确列出系统间需要对接的接口，是开发工作的直接依据。
*   **清单字段示例**：
    *   接口提供方
    *   接口调用方
    *   接口名称
    *   **接口状态**：**已有**（直接使用）、**新增**（需开发）、**已有修改**（需调整开发）。新增和修改均涉及开发工作量。

### 举例
*   **前端调用后端**：前端开发工程师调用订单系统后端，以实现“创建待支付订单”功能。
*   **订单系统调用库存系统**：订单系统调用库存系统接口，实现“锁定销售层库存”。

## 输入输出：定义数据交换内容
输入输出的数据结构定义了API接口的具体信息交换内容。
### 规则总结
*   **输入由接口提供方决定**：提供方需要什么数据来完成业务，这些数据就构成输入项。
*   **输出由接口调用方决定**：调用方页面需要展示什么信息或业务需要什么结果，这些就构成输出项。

### 场景分析
#### 1. 前端查询性页面（如商品列表页）
*   **输出决定**：由**页面展示内容**决定。例如，积分商城列表页需要显示“商品图片”、“商品名称”、“所需积分”、“销售单价”，这些就是接口必须提供的输出字段。
*   **输入决定**：由**查询条件**决定。例如，为了返回正确列表，输入项必须包括“会员编码”、“调用渠道”。

#### 2. 后端业务接口（如订单推送发货单给WMS）
*   **输入决定**：由**接口提供方（WMS）的业务需求**决定。WMS发货需要知道“收货地址”、“收件人信息”、“订单渠道”（影响包装）、“发货单号”、“发货仓库”、“商品明细”等，这些即是输入项。
*   **输出决定**：由**接口调用方（订单系统）的需求**决定。但在此场景下，输出可能是即时的“接收成功/失败”状态，而最终的“发货结果”则需通过**异步**回调接口通知。

## 总结
产品经理掌握API接口沟通的核心在于理解并运用三个层次的知识：首先通过**系统流程图**宏观把握跨系统协作逻辑与数据流向；其次基于流程图梳理出具体的**API接口清单**，明确各接口的状态与责任方；最后清晰定义每个接口的**输入输出**内容，遵循“输入由提供方业务需求定，输出由调用方数据需求定”的准则。掌握这套方法，便能有效避免技术沟通障碍，精准规划排期，提升产品与研发的协作效率。
