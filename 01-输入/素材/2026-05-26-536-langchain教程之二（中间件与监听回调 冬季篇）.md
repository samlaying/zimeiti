# langchain教程之二（中间件与监听回调 冬季篇）

> [](https://space.bilibili.com/) | [BV1TZfQBrEaa](https://www.bilibili.com/video/BV1TZfQBrEaa) | 时长 0:00

## 1. 模型回退中间件概述
模型回退中间件是一种用于在模型调用失败时自动切换到备用模型的中间件，提高系统的可靠性和容错能力。它可以处理多个模型之间的回退逻辑，确保即使某个模型不可用，系统仍能继续运行。

### 1.1 模型类型选择
- 模型可以是字符串或base chat model。
- 推荐使用base chat model，因为字符串类型无法处理复杂情况，例如会员大模型可能有额外参数（如deep sync没有），而base chat model更通用。

## 2. 实现步骤
### 2.1 导入必要的包
- 导入ChatDeepSick包：`from langchain.chat_models import ChatDeepSick`。
- 添加代码从环境变量读取API密钥：确保从文件中读取环境变量，避免硬编码敏感信息。

### 2.2 配置模型列表
- 设置多个模型，例如L1、L1DORI-RM3等。
- 具体例子：
  - 第一个模型：API密钥错误（故意设置错误）。
  - 第二个模型：模型名错误（例如使用一长串错误名称）。
  - 第三个模型：API密钥和模型名均正确（从环境变量文件中读取）。
- 这确保前两个模型会失败，第三个模型作为备用。

### 2.3 创建回退中间件对象
- 使用Fullback回退中间件类，传入模型列表：`Fullback = Fullback回退中间件(模型=[L1, L1DORI-RM3, 正确模型])`。
- 对象命名：例如`Fullback`，用于后续使用。

### 2.4 使用中间件
- 复制原有代码（例如agent执行链），并将回退中间件集成进去。
- 示例：在agent.2中加入call back参数，但为了调试暂不写call back。
- 添加塞（sleep）或延迟以确保错误信息可见。

## 3. 错误处理和调试
### 3.1 添加回调以监控错误
- 设置回调函数来打印错误信息，帮助识别问题。
- 错误会出现两次，对应前两个错误模型。

### 3.2 错误类型示例
- 第一个模型错误：API密钥问题（红色错误提示）。
- 第二个模型错误：模型不存在（模型名错误导致）。
- 系统自动跳过失败模型，找到第三个正确模型并成功执行。

## 4. 与其他中间件集成的问题
### 4.1 Summary总结中间件的限制
- 如果尝试与Summary总结中间件一起使用，总结中间件中的模型不会自动替换。
- 原因：总结中间件中的模型是硬编码的，回退中间件无法影响其内部模型配置。
- 结果：即使回退中间件替换了主模型，总结中间件仍使用初始设置的错误模型，导致持续失败。

### 4.2 解决方案建议
- **重写总结中间件**：继承原始Summary中间件类，并重写方法以加入模型判断逻辑，使其能响应模型替换。
- **专门写判断模型的方法**：创建一个独立的函数或模块，用于动态检查和替换模型，确保所有中间件使用正确的备用模型。

## 5. 建议和最佳实践
- **使用base chat model**：确保模型兼容性，处理复杂参数。
- **环境变量管理**：从文件或环境变量读取API密钥，避免硬编码。
- **测试与调试**：在实现时故意设置错误模型，通过回调监控错误，验证回退逻辑。
- **集成多个中间件时**：注意模型替换的一致性，必要时自定义中间件以处理替换逻辑。

总结核心要点：模型回退中间件通过配置多个模型并自动切换到可用模型来提高系统鲁棒性；实现时需导入正确包、设置环境变量、创建中间件对象并集成到执行链中；错误处理可通过回调监控；当与其他中间件如Summary集成时，需自定义解决方案以确保模型替换生效。
