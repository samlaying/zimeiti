# AI产品经理的 Vibe Coding 启蒙课

> [](https://space.bilibili.com/) | [BV13zo9B1E7M](https://www.bilibili.com/video/BV13zo9B1E7M) | 时长 0:00

## Vibe Coding 启蒙课笔记

### 一、 核心工具介绍：Cursor 中的三种模式
课程以 Cursor（以及类似的工具如 Tree）为例，介绍了其提供的三种主要工作模式，对应不同的能力和使用场景：
1.  **Chat**：仅有**读取**能力，不会修改项目中的任何文件。
2.  **Builder**：既能**读取**，也具备操作和修改本地文件的功能。
3.  **Builder with MCP**：在 Builder 的基础上，增加了用户自行配置的**各种 MCP（Model Context Protocol）** 服务的能力。

### 二、 如何连接 MCP 服务
MCP 是一套标准协议，允许 AI 助手（IDE）连接到外部工具和服务，以获取数据或执行特定任务。

#### **2.1 配置方法**
当选择 “Builder with MCP” 模式且未配置任何服务时，工具会提示添加 MCP。有两种主要方式：

##### **2.1.1 手动添加 MCP**
- **本质**：将一个符合 MCP 格式校验规则的 **JSON 数据**粘贴到指定位置。
- **操作步骤**：
    1.  从 MCP 服务商处获取对应的 JSON 配置代码。
    2.  在 Cursor 或 Tree 的 MCP 管理界面，选择“手动添加”。
    3.  将复制的 JSON 数据粘贴到输入框。
    4.  点击确认，即可完成配置。
- **注意**：不同 IDE（如 Cursor, Tree）的 JSON 字段结构完全一致，遵循 MCP 的标准。

##### **2.1.2 从市场添加 MCP**
- **操作步骤**：
    1.  在 MCP 管理界面，选择“从市场添加”。
    2.  浏览市场，选择所需的 MCP 服务（如 Figma MCP）。
    3.  根据提示填写必要信息（如 API Key），系统会自动生成或映射为 JSON 配置。
    4.  确认后即可添加。

### 三、 MCP 实战演示

#### **3.1 案例一：查询星座运势（手动添加）**
1.  **获取配置**：从火山引擎官网的 MCP 市场，找到“星座运势 MCP”，复制其在 Tree 下的 JSON 配置。
2.  **手动添加**：将 JSON 粘贴到 Tree 的 MCP 手动添加界面并确认。
3.  **调用测试**：在对话框中提出查询需求，如“查询天蝎座本周的运势情况”。AI 会自动触发对应的 MCP 调用，返回结果。

#### **3.2 案例二：基于 Figma 设计稿生成网页（从市场添加）**
1.  **添加 Figma MCP**：
    - 在 MCP 市场选择 “Figma MCP”。
    - 需要获取 **Figma API Key**。
    - **获取 API Key 步骤**：
        a. 登录 Figma，点击头像进入个人主页。
        b. 点击头像 -> `Setting`。
        c. 进入 `Security` 设置，找到 `Personal access tokens`。
        d. 生成一个新的 Token，设置名称和过期时间，并勾选所需权限。
    - 将生成的 API Key 填入 Figma MCP 的配置字段中，完成添加。
2.  **生成网页**：
    - 复制一个 Figma 设计稿的链接。
    - 在 Builder with MCP 模式下，要求 AI 根据该链接生成一个静态网页。
3.  **过程中可能遇到的问题及处理**：
    - **问题**：AI 调用 MCP 下载设计稿中的图片时可能失败。
    - **解决思路**：
        - **耐心与模型自调整**：首次失败时可不急于干预，观察模型是否会根据错误信息自行调整策略。
        - **更换模型**：若某模型（如 Claude 3.5 Sonnet）持续失败，可尝试更换其他模型（如豆包 1.6）。不同模型对 MCP 工具描述的理解和错误处理能力不同。
        - **明确指令**：在重试时，给出更明确的指令，如强调“使用 MCP 提供的图片下载功能，而非自己绘制”。
    - **版本回滚**：在 AI 流程中，任何时候都可通过对话历史的“返回”按钮，将整个项目的代码和文件回滚到该对话发起时的状态，这是一个安全的操作。

### 四、 AI 协作与 Vibe Coding 心得
在使用 Vibe Coding 进行产品开发时，以下心得至关重要：

#### **4.1 交互是双向的，而非单向命令**
- **原因**：与 AI 沟通不足会导致最终产品 99% 偏离预期。
- **做法**：在项目开始前，就与 AI 确立沟通节点，让它在可能走不同路径或面临选项时主动向你列出，由你参与共同决策。

#### **4.2 在产品完成前，先设计完整的测试用例**
- **做法**：思考产品功能点及其一般用例、边界用例，并将这些信息告诉模型。
- **益处**：
    1.  让模型更清楚预期结果。
    2.  模型可在每次迭代后自行运行测试，并根据结果决定是否需要继续迭代，减少人工全面审查的负担。

#### **4.3 结合项目，要求代码模块化**
- **问题**：避免将所有代码放在一个文件中，否则对模型的上下文管理和人类的阅读修改都是挑战。
- **做法**：要求模型将代码**拆分到不同文件**中，实现功能模块化，并在 `README` 文件中说明每个文件的用途。
- **益处**：
    - `README` 成为“字典”，便于快速定位需要修改的文件。
    - 修改某个模块时，只需获取该模块代码，减少对上下文的占用。
    - 保证模块间通过明确的输入输出接口交互，内部修改不会影响其他部分。

#### **4.4 持续学习，避免黑箱**
- **建议**：虽然 Vibe Coding 降低了编程门槛，但不应将代码视为黑箱。
- **做法**：在每个项目完成后，逐文档、逐行代码地与模型沟通，询问其逻辑和编写原因。
- **目标**：逐渐能够读懂代码，从而能在更细微的层面与 AI 合作，不断提高协作的上限。

### 五、 总结核心要点
本节课作为 AI 产品经理的 Vibe Coding 启蒙，核心在于**将 AI 编码助手与外部工具（MCP）连接，以扩展其能力**。关键步骤包括选择合适的工作模式（Builder with MCP），并通过手动或市场方式配置 MCP 服务。课程通过 Figma 设计稿转网页的实例，演示了完整流程以及遇到问题时（如图片下载失败）的调试思路。最后，课程强调了高效 AI 协作的四大心得：保持**双向交互**、**提前设计测试**、推动**代码模块化**以及坚持**持续学习**，以确保开发过程可控、结果符合预期，并不断提升人与 AI 协作的深度。
