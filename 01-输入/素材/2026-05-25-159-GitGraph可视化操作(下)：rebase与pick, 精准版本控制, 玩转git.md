# GitGraph可视化操作(下)：rebase与pick, 精准版本控制, 玩转git

> [__阿岳__](https://space.bilibili.com/480804525) | [BV1S1w3zMEii](https://www.bilibili.com/video/BV1S1w3zMEii) | 时长 41:24

## 一、rebase操作概述
- **目的**：让功能分支跟进主分支，避免产生额外的合并节点，保持GitGraph分支图简洁，尤其适用于多人协作场景。
- **与merge对比**：merge操作会产生合并节点，使分支图变复杂；rebase通过复制节点嫁接到主分支顶部，实现线性历史。

## 二、rebase的原理和操作
### 1. rebase的基本原理
- **原理**：找到功能分支与主分支的最近公共祖先，将功能分支的节点复制并放置到主分支的最新提交上。实际是复制操作，非移动；如果功能分支有子分支，子分支不会被移动。
- **例子**：功能分支（F1→F2→F3）rebase到主分支（M1→M2→M3→M4）顶部，变成M1→M2→M3→M4→F1'→F2'→F3'。

### 2. rebase操作步骤（在GitGraph中）
- **步骤**：
  1. 切换到功能分支（如feature）。
  2. 右键主分支标签，选择“rebase current branch on branch”。
  3. 点击Yes进行rebase，解决冲突（每个节点可能需单独解决冲突）。
  4. 强制推送到远程：右键功能分支，选择“push branch”，勾选“force with lease”（较安全）或“force”，点击Yes推送覆盖远程分支。
- **注意**：不要点击GitGraph中出现的“同步更改”蓝色按钮，否则会创建merge节点，破坏rebase效果。

### 3. rebase的潜在问题与建议
- **冲突解决**：rebase过程中可能需要多次冲突解决（每个节点一次），预期冲突多时，merge可能更合适。
- **远程分支同步**：rebase后本地分支与远程分支指向不同，必须强制推送更新远程分支。
- **协作建议**：如果分支有他人提交或已合并，避免rebase；单人单功能分支时安全使用。

## 三、cherry-pick操作
### 1. cherry-pick的概念
- **定义**：像摘樱桃一样，将某个提交节点从一个分支复制到另一个分支，保留原作者信息（author和committer）。
- **场景**：单独应用特定提交（如修复漏洞或接口变更）到其他分支，避免污染分支历史，利于git blame追溯。

### 2. cherry-pick操作步骤
- **步骤**：
  1. 切换到目标分支（如主分支）。
  2. 右键要cherry-pick的提交节点（即使无分支标签指向）。
  3. 选择“cherry-pick”，点击Yes。
  4. 解决冲突（如果有）。
- **例子**：将功能分支中的F3.txt提交cherry-pick到主分支。

### 3. 去除分支链中的特定节点
- **方法**：
  - 使用cherry-pick手动复制节点到一个新分支，然后删除原分支（适合少量节点）。
  - 或使用交互式rebase（适合长链条）。

## 四、交互式rebase（rebase -i）
### 1. 交互式rebase的用途
- **功能**：修改提交历史，如压缩提交（squash）、删除提交（drop）、重新排序等。
- **前提**：安装GitLens插件（免费部分可用），并在终端配置一次性指令（如`git config --global rebase.instructionFormat "[%h] %s"`）。

### 2. 交互式rebase操作步骤
- **步骤**：
  1. 在GitGraph中，右键要开始影响的提交节点（如分支起点），选择“rebase current branch on this commit”，勾选“开启交互式变基”。
  2. 点击Yes，在新的终端中进行操作（GitLens提供UI界面）。
  3. 使用下拉菜单选择操作：如drop（删除节点）、squash（向下压缩节点）。
  4. 点击“start rebase”完成，解决冲突（如果有）。

### 3. 压缩提交（squash）
- **软重置压缩最近提交**：
  - 命令：`git reset --soft HEAD~n`（n为提交数量），将最近n个提交压缩到工作区，然后重新提交。
  - 局限：只能压缩从后往前的提交，无法压缩中间部分。
- **交互式rebase压缩中间提交**：
  - 在交互式rebase中，使用squash操作将多个提交合并为一个，例如将B.txt、B1、B2压缩成一个节点。

## 五、多人协作中的rebase建议
- **使用`git pull --rebase`**：拉取代码时避免合并节点，保持分支线性。GitGraph中需手动输入命令（无直接UI操作）。
- **协作流程**：要求所有协作者使用rebase工作流，避免在分支上使用merge，否则无法继续rebase。
- **优势**：简化分支图，减少视觉干扰，提高代码审查和历史查看效率。

## 六、总结
视频详细讲解了Git中rebase和cherry-pick的操作、原理、场景及高级用法（如交互式rebase和提交压缩）。rebase通过复制节点线性化历史，cherry-pick实现精准提交控制，交互式rebase允许修改历史。这些工具帮助开发者在协作中优化版本控制流程，保持分支图清晰，提升效率。

**核心要点**：掌握rebase和cherry-pick是高效Git版本控制的关键。rebase用于线性化分支历史，避免合并混乱；cherry-pick允许选择性应用提交；交互式rebase提供精细历史修改能力。在协作中，合理使用这些工具可以简化分支图、提高代码管理效率，但需注意冲突解决和远程同步。
