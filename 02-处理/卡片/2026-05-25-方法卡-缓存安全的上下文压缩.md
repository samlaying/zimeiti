# 缓存安全的上下文压缩

当长对话的上下文窗口将满时，需要压缩历史。直接另起请求做压缩会浪费缓存和费用，应采用**缓存安全分岔**方法。

**问题**：另起压缩请求若使用不同系统指令或未带工具定义，将导致与主对话的前缀不匹配，缓存无法复用，且需支付完整的无折扣费用。

**解决方案（缓存安全分岔）**：
1.  **参数一致性**：压缩请求必须使用与主对话**完全相同的系统指令、用户上下文和工具定义**。
2.  **携带历史**：把主对话消息作为历史消息带上。
3.  **新增指令**：在消息末尾添加一条新的用户消息，内容为压缩指令。
4.  **预留缓冲区**：提前为压缩输出留出空间，不能等窗口完全填满才压缩。

**效果**：从后台视角看，该压缩请求的前缀与主对话请求高度相似，可直接复用缓存。新增成本仅为末尾那条压缩指令本身。
